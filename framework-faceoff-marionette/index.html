What is Backbone.Marionette?

Backbone.Marionette is a composite application library for Backbone.js that aims to simplify the construction of large scale JavaScript applications.

— marionettejs.com

Emphasis mine

Built on Backbone.js

So its a library for Backbone.js, what does that mean?

What is Backbone.js?

  • Models (and Collections)
  • View (and Template)
  • Router
  • 17,000+ Stars on Github
  • 6.4kb over wire
  • Tiny, read-able, well documented source
  • Models responsible for working with persistence APIs
  • Views handle interaction between Model/Collection and Template(Dom)
  • Templates are the DOM structure of the View
  • Routers handle proper screens are displayed based on url state

Backbone is event based

  • Models trigger Events
  • DOM interactions trigger Events
  • View manages events
  • No automatic bindings
  • Events are core to Backbone (and Marionette).
  • Model field updated, model saved.
  • User clicks something, fills form field.
  • Unlike Ember and Angular you must render/update DOM explicitly.

Backbone is great!

Backbone.js is small, flexible, and un-opinionated.

  • No surprises.
  • Built on the awesome Underscore library.
  • Great 3rd party libraries.
  • At under 1500 lines of code (with commets) Backbone is super easy to learn and use.
  • The _ way carries though everywhere here. No injection of Object or any unexpected behavior.
  • Tons of plugins, from Frameworks like Marionette to binding systems like Backbone.StickIt.

Backbone is a pain

Backbone.js is small, flexible, and un-opinionated.

  • Decide on (and implement consistently) your own best practices.
  • Write your own rendering code. A lot.
  • Handle switching between views.
  • Handle view clean up.
  • Handle event un-binding.
  • Organize your app in some sane way.

Frankly, I'm just not that good of a developer. I have an API to build, deadlines to hit, clients to please, servers to maintain. Not to mention co-workers to troll, a dog to walk and cats screaming for something! Trusting someone else to handle these common details allows me to focus my development effort 100% on the problem I am trying to solve. Both Angular and Ember handle these things for you, and so does Marionette.

Backbone.Marionette

  • Views & View Managers (Layouts, Regions)
  • Component based Modules
  • Eventing Tools
  • Marionette is a layer on top of Backbone.
  • These additions address many of the issues I have with plain Backbone.
  • Retains the flexibility, simplicity, and explicit nature of Backbone.
  • Lots to cover hear, going to blow through

Marionette Views & View Management

Marionette.View

  • Inherits from Backbone.view.
  • Adds standard templating and rendering.
  • Implements a standard API for "closing" a view.
  • Is prototype of every other Marionette view type.
  • ItemView extended from Backbone.View, has same functionality + more.
  • Template can be a selector to dom or function. Compatible with Handlebars and Mustache, as well as the default Underscore.

Marionette.ItemView

  • Renders one model.
  • Used for forms, display, listing items.
  • Can be bound to render on model events such as change.

            <div id='app'></div>
            <script id='item-view' type='text/html'>
              <h2><%= name %></h2>
              <p><%= tagline %></p>
            </script>
          

            var Framework = Backbone.Model.extend({});
            var FrameworkView = Backbone.Marionette.ItemView.extend({
              template: '#item-view',
              modelEvents: { 'change' : 'render' }
            });
            var framework = new Framework({name: 'marionette.js'});
            var view = new FrameworkView({model: framework});
            $('#app').html(view.render().$el);

https://github.com/marionettejs/backbone.marionette/blob/master/docs/marionette.itemview.md

  • Provides events/hooks on rendering/closing the view.
  • Provides easy way to add custom functionality based on model events.

Marionette.CompositeView

  • Renders an optional Model and a Collection, renders each Collection item in an Item view.
  • Great for Model + Related Collection and Leaf view structures.
  • Automatically keeps dom in sync with Collection.

            <script id='frames-view' type='text/html'>
              <h1>Frameworks:</h1>
              <div class='frameworks'></div>
            </script>
          

            var Frameworks = Backbone.Collection.extend({model: Framework});
            var FrameworksView = Backbone.Marionette.CompositeView.extend({
              template: '#frames-view',
              itemView: FrameworkView,
              itemViewContainer: '.frameworks'
            });
            var frameworks = new Frameworks([framework]);
            var compView = new FrameworksView({collection: frameworks});
            $('#app').html(compView.render().$el);
  • I am skipping Collection view, which is useful but I almost always use a Composite view instead.
  • Composite views are incredibly helpful, and can dry up backbone code significantly.

Marionette.Layout

  • Manages opening and closing of child views.
  • Has an instance of Marionette.RegionManager built it.

            <script id='frames-layout' type='text/html'>
              <div id='menu'></div>
              <div id='main'></div>
            </script>
          

            var FrameworksLayout = Backbone.Marionette.layer.extend({
              template: '#frames-layout',
              regions: {
                main: '#main',
                menu: '#menu'
              }
            });

            var layout = new FrameworksLayout();
            $('#app').html(layout.render().$el);

            layout.main.show(compView);
            layout.main.close();
  • This is a common pattern and problem in backbone apps, and these layouts are a very elegant solution.
  • These tie in particularly well to the modules piece
  • So, just this piece of Marionette can DRY up most backbone apps.

Composite Applications with Marionette

Backbone.Wreqr

Event Aggregator


            var vent = new Backbone.Wreqr.EventAggregator();
            vent.on('foo', function() {
              console.log('foo event');
            });
            vent.trigger('foo');

RequestResponse


            var reqres = new Backbone.Wreqr.RequestResponse();
            reqres.setHandler('foo', function() {
              return 'foo requested. this is the response';
            });
            var result = reqres.request('foo');
            console.log(result);
  • Part of Marionette
  • Tools to extend the Event based architecture of Backbone to a higher level
  • Event Aggregator - Applicaiton/Module level events
  • Request Response - Allows decoupling and abstract APIs for calling functions cross module

Marionette.Application


            var App = new Backbone.Marionette.Application();

            // No Template, uses HTML as rendered (by server).
            App.addRegions({
              app: '#app',
              nav: '#nav'
            });

            App.addInitializer(function(options) {
              Backbone.History.Start();
              App.vent('history:started');
              App.reqres.setHandler('foo', function() { return options.foo; });
            });

            App.start({foo: 'bar'});
  • Applications are where things come together.
  • Typically you just have 1 of these.
  • Use some module system, or attach app to window.
  • Includes region manager (similar to layout, but uses existing DOM for template
  • Includes Event Aggregator
  • Includes Request/Response
  • Contains all the logic to start up your app.

Marionette.Application.Module

Isolating Models, Maintaining State


            App.module('Entities.Frameworks', function(Module, App) {
              var Model = Backbone.Model.extend({});
              var Collection = Backbone.Collection.extend({model: Model});

              // Manage loaded state (fetch logic omitted here);
              var collection = new Collection();

              App.reqres.setHandler('framework:collection', function() {
                return collection;
              });

              App.reqres.setHandler('framework:model', function(id) {
                return collection.get(id);
              });
            });
  • Modules are tools to isolate your code in related sections.
  • Similar to what you get with Require.js, but very oriented to the app/module/sub module pattern.
  • Plays okay with require.js, plays very well with Browserify
  • In this example we have added a model(and sub module) that manages collection state.
  • In the real world there would probably be some logic here to fetch the collection, which could be isolate here.

Marionette.Application.Module

Isolating Screen Logic


            App.module('Frameworks'); // Typically has routing logic
            App.module('Frameworks.Show', function(Show, App) {
              this.startWithParent = false;

              var ItemView = Backbone.Marionette.ItemView.extend({
                template: '#item-view',
                modelEvents: { 'change' : 'render' }
              });

              var CompositeView = Backbone.Marionette.CompositeView.extend({
                template: '#frames-view',
                itemView: FrameworkView,
                itemViewContainer: '.frameworks'
              });

              this.on('start', function() {
                var collection = App.reqres.request('framework:collection');
                var view = new CompositeView({collection: collection});
                App.main.show(view);
              });
            });
  • In this example we are isolating our view logic.
  • Modules are Event Aggregators

Managing Large Applications

  • One Application
  • Model Modules
    • Request-Response API
    • Manage model state
  • One Module Per "Section"
    • Manages what screen is displayed
    • Home to the Router for that section
  • One Sub-Module per "Screen"
    • Manages Layout for screen
    • Renders appropriate components in regions
    • Event aggregate for cross component interactions
  • One Sub-Module per "Component"
    • Renders views
    • Triggers events up the chain
  • We have seen the tools, but how your organize is extremely important

Marionette VS The Competition

  • Super explicit, super simple.
  • Focused on OO concepts such as composition and de-coupling.
  • Manual binding of data and events.
  • Must manage routes and state explicity.
  • Much more straight forward than Ember.js.
  • More model focused than DOM specific frameworks like React.
  • More code required than Ember or Angular, but much less then Backbone.

Should you use Marionette?

Maybe.

  • It isn't right for every project.
  • Large scale.
  • Value flexibility over convention.
  • Value explicit interactions over magic.
  • Value extremely good documentation.
